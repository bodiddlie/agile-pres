Customer Interaction
  • Customer should be involved throughout project
  • Building requirements through user stories is a customer centric activity
  • Requirements are expected to change
  • Continuous Improvement
User Stories
  • Working with the customer to determine not just requirements, but also design
  • Deep dive into what the customer wants
  • Catch issues/concerns early
  • Get that crucial customer buy-in
Example User Story

As an Engineering Customer I need to be able to sign-in at the front desk.

Success Criteria
  • Can enter name and reason for visit
  • Both fields are required
  • Dropdown for common reason choices
  • After submission, staff are notified of customer in queue
Breaking Story Into Tasks
  • Create Model for customer entry
  • Create Model for reason suggestions
  • Seed data for reason suggestions
  • Build interface for entering data
  • Build backend API for receiving data
  • Configure notifications
Project Estimation
  • Estimation is difficult
  • Current process is a best-guess effort
  • Not tracking our estimates
Better Estimates
  • Break User Stories into achievable tasks
  • Estimate effort, not time
  • Assign points to reflect amount of effort (e.g. 0, 1, 2, 3, 5, ...)
  • Group tasks into sprints
  • # of points finished in sprint = velocity
Iterations
  • Specific time interval. Same every time
  • Every sprint results in a deliverable piece of software
  • Using velocity we determine what features will fit in a iteration
Iterations & Deadlines
  • Meet deadline exactly? Awesome, onto the next.
  • Come in under the deadline? Good, recalculate velocity, onto the next.
  • Cut features or push deadline? Fine. Evaluate why, onto the next.
The key

They key is to constantly be reevaluating the tasks to see where they fit in the project. Tracking this gives us meaningful data on what we're doing and how we can change to make it better.

Dealing With Defects

Issues found as a result of working on a sprint are not defects. They are part and parcel of development. They are to be fixed immediately and then they are done. Time spent tracking these is time lost focusing on the sprint.

Once a sprint is finished and a deliverable is out, any defects found or reported must be tracked. These will often become new tasks or stories that need to be evaluated.

Tracking Defects

Tracking defects that are found out-of-sprint is very important. They need to be carefully analyzed to determine why they made it out of sprint. This is not negative tracking. It just means we can potentially learn from it and prevent it during sprints in the future.

Code Review

At the end of every iteration, the team gets together to take stock of what was accomplished. Part of this is conducting code reviews. Each developer will present and go over the code that is unique to their project. The focus here is to evaluate the application specific bits of design and code for maintainability. Correctness should be already validated via unit tests and code linting.
